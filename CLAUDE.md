## Working Rules

**Before starting any task:**
- Read docs/PRD.md for scope and intent.
- Ask clarifying questions until 95% confident. Do not make any assumptions. For each question, do research and try to answer yourself first before presenting the questions to user.
- For features and non-trivial fixes, follow the `/deep-feature` pipeline (skill routing ‚Üí audit
  ‚Üí research ‚Üí opinion ‚Üí Codex adversarial review ‚Üí plan ‚Üí implement). Trivial = non-behavioral
  text/formatting only ‚Äî state why before skipping.

**After every major step:**
1. Run `flutter analyze` ‚Äî zero issues required.
2. Run `flutter test` if tests exist.
3. Always screenshot the affected UI section.
4. Review: theme coherence, design consistency, functionality correctness.
5. Fix all issues before proceeding.

**Always:**
- One change at a time. Never rewrite whole files unless explicitly asked.
- State after each change: what changed, which file, what to test.
- Never build features outside docs/PRD.md without asking first.

## Architecture

RPG-gamified workout tracker. Flutter app, Android-only. All persistence uses `SharedPreferences` with JSON serialization (no SQLite in practice, despite drift being a dependency).

### Directory layout

- `lib/pages/` ‚Äî screens. Workout session pages live in `lib/pages/Workout session/` (note the space). Onboarding screens in `lib/pages/onboarding/`.
- `lib/services/` ‚Äî business logic. Each service owns its own SharedPreferences key(s). Services are instantiated ad-hoc (no DI container), often with injectable `nowProvider`/`catalogOverride` for testability.
- `lib/models/` ‚Äî data classes. All JSON-serializable with `fromJson`/`toJson`.
- `lib/data/` ‚Äî static constants and registries: `curated_exercises.dart` (per-muscle-group exercise ID allow-lists), `class_definitions.dart` (class‚Üímuscle mapping), `loot_registry.dart` (all loot items), `muscle_groups.dart` (7-bucket canonical taxonomy), `programs_library.dart`, `strength_standards.dart`.
- `lib/theme/` ‚Äî `tokens.dart` (single source of truth for palette, spacing, motion constants), `app_fonts.dart` (local `ShareTechMono` font helper).
- `lib/widgets/` ‚Äî reusable arcade-themed components. `widgets/motion/` has micro-interaction wrappers (hold-depress, phosphor-tap, ambient-drift, etc.).

### App boot sequence (`main.dart`)

1. `MigrationService.runOnce()` ‚Äî cleans dead SharedPreferences keys from removed features.
2. `ClassMigrationService().migrateIfNeeded()` ‚Äî auto-assigns class for existing users.
3. `MigrationService.runFeatureGateSeedOnce()` + `FeatureGateService().load()` (via `BootService`) ‚Äî
   grandfathers existing installs into the earned feature-unlock ladder and seeds the sync gate
   snapshot before the shell's first frame (no lock-state flash).
4. `_AppGate` ‚Äî checks `OnboardingService().isComplete()`: routes to `OnboardingFlowPage` or `RootPage`.

(There is **no stat-decay step** ‚Äî earned stats are immutable; the old `applyDecayIfNeeded` was removed.
See Combat Stats below and `docs/stats-mechanics.md` ‚Üí "Decay (removed)".)

### Navigation structure

`RootPage` (`lib/pages/root_page.dart`) is the shell: a bottom bar of **four** browseable destinations ‚Äî `enum AppDestination { home, inventory, guild, labs }` ‚Üí **Home ¬∑ Items** (Inventory) **¬∑ Guild ¬∑ Labs** ‚Äî flanking one elevated **center TRAIN action**. TRAIN is **not a destination/tab**: it's `TrainNavButton`, whose `_onTrainTapped` resumes a live/paused session, opens/commits the **in-shell exercise-selection draft**, or opens today's mission. The four destinations render through an `IndexedStack` of `HomePage`, `InventoryPage`, `GuildPage`, and `ProfilePage` ‚Äî **Labs** hosts `ProfilePage` (identity/stats/settings). Home, Guild, and Labs each carry a `GlobalKey` for refresh-on-return; Inventory self-reloads on init.

The dropped **Workout** tab's two halves are now **push** surfaces (in `lib/pages/workout_page.dart`), not tabs: workout **history/calendar/analytics** is `WorkoutLogsPage` (pushed from Home's last-workout card and from Labs), and **Programs ‚áÑ Exercises** is `WorkoutLibraryPage` (pushed from Labs). **Quests** is likewise a push ‚Äî from Home (`onViewQuests`) and the home-room board ‚Äî not a tab. The gem **Shop** also pushes from Home. **Strength progression is the body map itself** (Concept #1, "the body is the browser"): tapping a muscle opens its **strength dossier** (`_MuscleDossierSheet` in `muscle_body_map.dart`) ‚Äî a header coverage verdict for THIS WEEK (the glance isn't lost) over the muscle's **strength roster**: the lifts that train it (filed under their **primary** muscle only ‚Äî `strengthByMuscle` in `body_map_regions.dart`, so a bench doesn't clutter the triceps roster), each a `StrengthMomentumRow` ‚Üí the existing `ExerciseHistoryPage` chart. Rows lead with a **plain verdict** (`NEW BEST`/`ON THE RISE`/`HOLDING`/`REBUILDING` ‚Äî never "e1RM" jargon; `REBUILDING` is the honest, non-punitive down state, never red), a signed "vs last" delta, and an **"est. max"** (an estimate, never called "best"). Data: `StrengthTrendService` (all-time momentum from `ProgressiveOverloadService.epley1RM` ‚Äî the same estimate the chart plots; `StrengthMomentum` band ¬±2.5%). A secondary **ALL LIFTS** route on the body map opens `StrengthIndexPage` (`lib/pages/strength_index_page.dart`) ‚Äî the completeness net (every weighted lift once, searchable) for the "show me everything" intent, reworked into a **visual roster**: each lift is a `StrengthRosterRow` (`widgets/strength_roster_row.dart`) ‚Äî a movement-pattern **lift icon** (`widgets/lift_icon.dart` + `data/lift_icons.dart`, a name-keyword classifier ‚Üí 13 pixel glyphs in `assets/icons/lift-icons/`, recoloured via `BlendMode.srcIn`) + the lift name + a big **est-max** number + a verdict **glyph** (‚òÖ new-best / ‚ñ≤ rising / ‚Äì holding / ‚ñº rebuilding, the down state muted never red) + a small signed delta ‚Äî text traded for visual identity. Lifts group into **completeness-preserving sections** (NEW BESTS / RECENTLY TRAINED / REBUILDING, every lift in exactly one, empty sections dropped); **filter chips** (ALL / RISING / NEW / REBUILDING) slice by momentum into a flat view; a single **`EST. MAX ¬∑ <unit>`** column hint carries the honest-estimate framing; a one-time reduced-motion-safe entrance stagger; only a genuine new best gets the amber flourish. A user can **pin up to 3 "anchor" lifts** (`PinnedLiftsService`, key `pinned_lift_ids_v1`, block-and-tell at the cap ‚Äî a 4th pin shows a floating "unpin one first" notice, never auto-evicts) ‚Äî pinned lifts surface as a richer **`PinnedLiftCard`** (`widgets/pinned_lift_card.dart`, cyan accent, the lift icon + verdict WORD + signed delta + neutral "trained N√ó" mastery line + the **one** sparkline with an amber PR marker) at the **top of the ALL view only** (filter views stay truthful), pulled out of their section so nothing shows twice. Pin via **long-press** a row (also a custom Semantics action for switch/screen-reader; a persistent "PINNED N/3 ¬∑ hold a lift to add" status line keeps it discoverable); unpin via the card's pin icon. Stale pins (a lift whose sessions were deleted) self-heal on load (`pruneTo`) so a ghost pin can't deadlock the cap. The **dossier** keeps the denser `StrengthMomentumRow` (muscle-grouped context). The inline Logs "LOAD TRENDS" top-3 (heaviest-weight, ‚â•5 sessions) is unchanged ‚Äî a complementary no-estimation glance. **Coverage shading/meter stays singular** (weekly sets); strength lives only in the dossier (no strength-lens ‚Äî avoids conflating two meanings in one brightness channel). The detailed per-muscle meter rows are **collapsed by default** per group (`_expandedGroups`) ‚Äî the page opens to the body + the group titles (SHOULDERS ¬∑ ARMS / CHEST ¬∑ CORE / LEGS) with an expand chevron; tap a header to reveal its rows (reduced motion skips the `AnimatedSize` rather than zeroing its duration). The Logs **session list** shows the 3 most recent, with **SHOW MORE** (+5 each) and a **SHOW LESS** (appears once expanded ‚Üí resets to 3).

**Workout session flow** (all in `lib/pages/Workout session/`):
1. `StartWorkoutPage` ‚Äî muscle group chips, time picker, exercise picker bottom sheet (multi-select with favorites). Primary entry is **in-shell** as the center TRAIN action's selection surface (`embedded: true`, survives tab nav); also pushed standalone (e.g. the empty-log CTA). A live **"today's targets" preview** (`widgets/target_body_preview.dart`) sits above the loadout: a compact static front/back pixel body that lights the muscles the selected lifts will train ‚Äî **primary bright, secondary dim** (mirroring coverage's 1.0/0.5 credit, never overstating a synergist) ‚Äî via the shared `targetedBodyMuscles()` mapping in `body_map_regions.dart` (same primary‚Üídetailed‚Üíbody path as the history coverage map, so they can't disagree). A plan **preview**, not a coverage verdict: binary intent only ‚Äî no MEV/MAV/zone words; only the trained side(s) render (a push day shows no all-dark back); the `TARGETS: ‚Ä¶` text line is the screen-reader source. Updates live as the loadout changes (research: the evidence-favored placement over enriching the confirm dialog ‚Äî the start is a forward-looking planning moment, so a partial highlight reads as intent, not the peak-end "incomplete" hazard of a post-workout map).
2. `ActiveWorkoutPage` ‚Äî live timer, per-exercise status tracking, rest timer. Tapping exercise pushes `ExerciseSessionPage`, awaits `List<SetEntry>`. All-done enables "Finish Workout". Each logged set silently checkpoints the ongoing session to storage (`checkpointOngoingSession`, no change-signal) stamping `WorkoutSession.lastActivityAt` + a credited `actualDurationSeconds` ‚Äî so a force-kill no longer loses logged sets. **Idle auto-save:** after 30 min (`WorkoutStorageService.idleTimeout`) with no new set, a reveal (`showIdleSessionDialog`, SAVE & FINISH / KEEP TRAINING / DISCARD) is offered ‚Äî by the active page's own timer while it's on top, or by `RootPage._showIdleRevealIfNeeded` on the next open/resume after a kill (gated on `ModalRoute.isCurrent`; both arbitrated by `IdleSessionGuard`). SAVE credits time only up to the last set; a zero-set idle session is dropped silently.
3. `ExerciseSessionPage` ‚Äî set logging (weight + reps). **Sequential gate:** only the first un-logged row (the *frontier*) is editable; later rows are gated (disabled fields, muted **empty-circle** `radio_button_unchecked_sharp` button) and a tap warns "Log your previous set first" (the opaque row `GestureDetector` catches it ‚Äî disabled `TextField`s ignore pointers). Logging a row fills its **empty circle ‚Üí filled `check_circle_sharp`**, starts the between-set rest, and shows a brief floating **"Rest timer started"** notice; re-tapping a logged row unlocks it (and unfocuses, re-gating the rows below). "Finish Exercise" validates and pops data back. (The one-time *plate-calc hint line* was removed; the plate-calculator suffix icon + sheet stay.)
4. `WorkoutSummaryPage` ‚Äî post-workout XP awards, stats, calorie breakdown. `PopScope(canPop: false)`. "Save & Exit" persists via `WorkoutStorageService` then pops to root. Opens on the **BIT Session-Complete ceremony** (`widgets/companion/session_ceremony.dart`, ported from `assets/design_handoff_session_ceremony`): BIT owns the screen 2.55s ‚Äî dormant ‚Üí inhale (haptic double-blink) ‚Üí amber surge (chime + shake) ‚Üí a 1.5s banked flight (dash-fwoosh SFX + amplitude-gated haptic swell) ‚Äî then lands (impact thud + haptic thump + puff) in a **72px seated `BitCompanion`** that permanently replaced the medal icon (hero rule: level-up holds CHEER, else NEUTRAL; tap = press orbit). The staged reveal is gated on `max(saveDone, touchdown)`; tap skips idempotently; reduced motion / abandoned / calibration get no ceremony (the seat appears settled). Each beat then enters via a **nature-fit entrance vocabulary** (all private to the page, layout-reserved, reduced-motion-static): the XP label slams (instant + punch-scale + amber strobe); ALL plain text types out (one text idiom ‚Äî title, potion row, receipt strip, BREAKDOWN heading, BIT's sign-off "Good haul today, warrior."); the XP receipt *prints* row-by-row with a TOTAL flash; the cache card *drops in* (stepped fall + squash + tier flash + dust); the charge card *powers up* (stepped brightness + double "charged" neon blink); the warm-up card *kindles* (amber wash cools, ember glow settles); reward badges get the *unlock stamp* (scale-slam + amber flash + 6-spark pixel burst); the stat-gains row is instant (its count-up roll IS the entrance); breakdown rows keep the per-row phosphor blink.

**Onboarding flow** (`lib/pages/onboarding/`): multi-screen cinematic sequence ‚Äî cold open ‚Üí problem ‚Üí solution ‚Üí calibration quiz ‚Üí name (creates the character) ‚Üí class reveal ‚Üí generating ‚Üí start gate. There is no avatar step: a gender-seeded default pixel face (`AvatarDefaults.forSex`) is assigned at name-commit and shown at the start gate; users edit it later from Profile. Completes by pushing `RootPage(openWorkoutStarterOnLaunch: true)`. The **name screen** opens with a BIT-presented **starter-readout panel** (`widgets/onboarding/starter_readout_panel.dart`) ‚Äî a faced, living `BitMoodCore` delivering a BIG "Your program is built, warrior." over a *reversible* recap of the recommended plan (class flavor ¬∑ program ¬∑ training days; weight/sex stay private, body-neutral; the whole card taps back to program selection to edit) ‚Äî so the user reads/validates the plan their answers built right before the irreversible name commit. Framed as a recommendation, **not** an owned identity (research: a derived class hard-labelled "YOUR BUILD" reads as imposed; insights.md). The screen's upper content is a `SingleChildScrollView` that lifts the focused field above the keyboard (`ensureVisible` on focus), since the panel pushes it down.

### Product doctrine

Every logged workout should make the user's character feel harder to abandon. Real training is the
fuel; identity, streak, rank, loot, and ritual are the psychological engine. When adding or changing
features, prefer surfaces that strengthen one of the long-term hooks:

- **Identity attachment:** avatar, name, class, rank, title, frame.
- **Competence growth:** stats, grades, XP, suggested loads, visible deltas.
- **Collection desire:** deterministic frames, themes, titles, future cosmetic horizons.
- **Ritual return:** home mission, workout summary, weekly cadence, LCK, guild signal.
- **Recovery protection:** rest days, shields, and VIT recovery that protect the build between
  sessions. Earned stats are immutable (gains never decay) ‚Äî the build is preserved by design, not
  threatened into preservation.

### Exercise data pipeline

1. `assets/exercises.json` ‚Äî **132 curated exercises** (trimmed from the original ~870-entry free-exercise-db down to the curated picker set; the search-only long tail and its image folders were removed via `ops/remove_exercise.py --from-file`) with `id`, `name`, `level`, `images`, `primaryMuscles`, `secondaryMuscles`. The `Exercise` model carries `primaryMuscle` (first primary) **and** `secondaryMuscles` (synergists, e.g. bench ‚Üí triceps/front delts; ~69% of built-ins non-empty, custom exercises have none); both feed **display/coverage analysis only** ‚Äî no volume/stat/XP/overload path reads them.
2. `ExerciseCatalogService` ‚Äî merges built-in (cached) + custom exercises. Single access point.
3. `data/curated_exercises.dart` ‚Äî `curatedExerciseIdsByMuscleGroup` map filters the picker to a curated subset per muscle group.
4. `data/muscle_groups.dart` ‚Äî canonical 7-bucket taxonomy (Chest/Back/Shoulders/Arms/Legs/Core/Full Body). `muscleGroupForDetailed()` maps raw muscle names from JSON ‚Üí buckets. `services/muscle_coverage_service.dart` is the pure analyzer behind the Logs **muscle-coverage body map**: `weeklySetsByBucket` rolls logged working sets into the **7 buckets**, `weeklySetsByMuscle` into **detailed muscles** (biceps‚âÝtriceps etc.) ‚Äî both fractional (direct primary = 1.0, indirect secondary = 0.5, primary wins on overlap; warm-ups/partial/unknown excluded). Detailed keys are the raw free-exercise-db tokens **except** the two that lack head granularity, split via `data/muscle_splits.dart` (`splitDetailedMuscle` + the curated `curatedMuscleSplits` override map, EMG movement-pattern rule): `shoulders`‚Üí`front_delt`/`rear_delt`, `abdominals`‚Üí`rectus_abdominis`/`obliques`, scoped to curated lifts ‚Äî an un-curated splittable token stays the **coarse generic token** (folded to front-delt/rectus by the map, never guessed onto rear/obliques). A `muscle_split_test.dart` integrity test validates every override against the real catalog. The **body map** itself ‚Äî `widgets/muscle_body_map.dart` (a faithful port of the `assets/body_diagram` handoff: a front/back pixel body whose muscles brighten by weekly sets, intensity via `Image`'s `opacity:` over alpha-only region masks tinted to one uniform `kCoverageLit` in code (`ColorFilter.srcIn` ‚Äî the baked per-muscle hue is discarded, so a recolor is a single-token edit), over a `kCoverageScrim`@0.5 base-dim, `RepaintBoundary`, read-only per-side meter, reduced-motion-static) driven by `data/body_map_regions.dart` (16-muscle region model + the ported ramp + mask‚Üîmuscle map) ‚Äî **replaces the old Muscle Balance bars** in `WorkoutLogsPage`. (The meter total still uses the shared `creditPerSet` ‚Üí `muscleBreakdown` rollup so the bar can't disagree with the coverage crediting; `weeklyContributors` remains the coverage path. The strength roster is a *separate* primary-muscle grouping ‚Äî `strengthByMuscle` ‚Äî answering "my lifts for this muscle", not "what worked it this week".) A **range selector** (`CoverageWindow` enum: `7-DAY` / `4-WK AVG` / `12-WK AVG`, periodization-length rolling windows, default `7-DAY`) sits above the body: the longer presets show the **weekly AVERAGE** (`MuscleCoverageService.averagedContributors` ‚Äî the single pure calc boundary; `_LogsTab` owns the window + recompute, the widget stays presentational), so a multi-week window stays comparable to the **weekly** MEV/MAV bands. The divisor is **capped to real history** (`min(window, now‚àífirstSessionEver)`, ‚â•1 wk) so a new user's hard week is never divided by empty pre-history weeks (reads RESTED); the unit is labelled **"avg/wk ¬∑ last N wk"** (the real span) so an average is never misread as raw recent work; longer views are opt-in + body-neutral (no scold for a deload-dimmed muscle).
5. Exercise images: `assets/exercises/exercises/<ExerciseName>/0.jpg`, `1.jpg`, etc.
6. A **batch** form (`--from-file <ids.txt>`) removes many in one transactional pass (refusing any program lift). **Dry-run by default** (prints a unified diff); **transactional** on `--apply` (stages deletions to a backup dir, rolls back on any failure); **refuses** a program lift without `--replace-with`; **warns** (never auto-edits) when the id is in `demo_seed_service.dart` (bespoke seed weights). It never touches persisted user data ‚Äî history snapshots the exercise *name*, favorites filter to the live catalog, pins self-heal ‚Äî so a removed exercise can't corrupt a log or crash the app (nothing force-resolves an id). `test/catalog_integrity_test.dart` is the umbrella net: every id referenced by curated / programs / `curatedMuscleSplits` / the demo manifest must resolve to a live catalog entry (an incomplete removal fails loudly in one place), every `pubspec.yaml`-declared image folder must still exist on disk (catches the build-breaking dangling declaration), plus per-day program invariants (unique suggested ids; prescription keys == suggested ids).

### Persistence pattern

All state in `SharedPreferences` as JSON strings. Key services and their storage keys:
- `workout_sessions` ‚Äî `WorkoutStorageService` (completed/ongoing sessions)
- `combat_stats` ‚Äî `StatEngine` (cached stat values)
- `quest_state_v1` ‚Äî `QuestService`
- `rest_state_v1` ‚Äî `RestService`
- `guild_v2` ‚Äî `GuildService` (guild entity; old `guild_v1` / `guild_members_v1` / `guild_forge_nods_v1` are purged by `MigrationService`)
- `loot_inventory` / `equipped_loot` ‚Äî `LootService`
- `active_character_v1` ‚Äî `CharacterService`
- `gem_ledger_v1` ‚Äî `GemService` (append-only, idempotent-by-id gem ledger; sources: quest / adventure / demoTopUp / cosmeticPurchase / **warmup**)
- `simple_mode_enabled_v1` ‚Äî `SimpleModeService` (opt-in **Simple Mode** toggle in Profile ‚Üí Settings, default OFF). When on it strips the *pre-workout scaffolding* at render time: the warm-up advisory card + the **TRY**/suggested-load prompt (`exercise_session`, read at screen init beside `_progressionEnabled`), the **curated first-run** loadout default (`start_workout._defaultIdsForGroup` skips the curated fallback ‚Äî history-derived defaults stay), and the progression re-opt-in nudge (`workout_summary`). It is **not** a product fork ‚Äî the in-set Set-1 auto-copy and the whole identity/XP/loot/class layer are untouched (research 2026-06-28: serve experienced users with adaptive defaults inside ONE app). In Settings, the granular "Suggested loads" row dims to "Off while Simple Mode is on" so there's one clear owner (Codex F2). **Also surfaced in onboarding** (2026-07-14) as a self-contained **"Workout guidance"** card on `RemindersPrimerPage` ‚Äî a reversible **Compact / Extra suggestions** choice (writes this same key; NOT a mode fork), **pre-selected** from the quiz's `Experience` via `simpleModeDefaultForExperience` (intermediate/advanced ‚Üí Compact/ON; novice/beginner ‚Üí OFF) with a tap-to-reveal preview mock and a "change anytime in Settings" line. The default is persisted **only when the card is displayed** (never at the earlier character-commit ‚Äî Codex F1: never silently reduce a first workout before it's been shown) and on every flip (single source of truth ‚Äî Codex F2); the guidance commits by its **own** controls, decoupled from the primer's TURN ON / NOT NOW notification buttons (Codex F4). A kill before the card is shown fails safe to OFF (more guidance, never less).

**Warm-up reward (re-anchored to logged sets):** warm-up sets are real, logged `SetEntry`s stored apart from working sets in `ExerciseLog.warmupSets` (working `sets` stays the volume/stat/XP/overload truth ‚Äî every aggregator reads it untouched). `WorkoutSession.warmedUp` is a **derived** getter (`exercises.any((e) => e.hasWarmupSet)`), not a stored field. Logged in the exercise screen via the advisory card's **SAVE** (one tap) or a manual **+ WARM-UP SET** in the warm-up sub-section; the session page round-trips a single `isWarmup`-flagged list, split into `sets`/`warmupSets` once at `_ActiveWorkoutPage._buildExerciseLogs` (resume recombines `[...sets, ...warmupSets]` so a force-kill never drops them). On save, `WarmupRewardService.grantForSession` (called from `saveSession`, like the Adventure charge) awards a small gem bonus via `GemService.awardWarmupGems` ‚Äî idempotent by **day** (`warmup:<dayKey>`), so it's capped at one/day, gated to a non-abandoned session with a real working set, and feeds no XP/stats. The optional unrewarded mobility guide is `data/warmup_routines.dart` (RAMP, tailored by `targetMuscleGroups`) shown by `widgets/warmup_sheet.dart`, reachable from Start.

`WorkoutStorageService.changes` is a broadcast `StreamController<void>` that notifies listeners when sessions are written ‚Äî `RootPage` subscribes to refresh quest-aware tabs.

## Haptics, Sound & Taps
- Every tappable routes through a **haptic-aware wrapper** ‚Äî `ArcadeTap` / `HoldDepress` / `PixelButton` / `ArcadeChip` ‚Äî so haptics wire **automatically by role** (button‚Üí`tap`, chip‚Üí`selection`; the low-level `HoldDepress`/`ArcadeTap` default silent for decorative taps). `HapticService` owns the semantic intents (`selection`/`tap`/`success`/`reward`/`warning`); `HapticPulseTrack` couples a pulse-train to an animation.
- **Never** a raw `GestureDetector(onTap:/onLongPress:)` or `InkWell` for a tappable ‚Äî `test/tap_haptic_coverage_test.dart` fails on any outside the wrappers; a genuine raw gesture (drag/pan/decorative/custom hit-area) needs an inline `// haptic-ok: <reason>`. The test's `_baseline` (31 pre-existing files) is **shrink-only** ‚Äî migrate down, never add.
- Intensity discipline: subtle `selection` broadly, heavier `success`/`reward`/`warning` reserved for confirm/reward/destructive; **no drones** (drag haptics fire on quantized step-crossings, not per-frame; reward peaks ~400ms). Reduced motion does **not** mute haptics (they carry their own Settings toggle).
- **Interaction-tier SOUND** rides the same seam: `PixelButton` maps its haptic intent to a pooled low-latency sound (`tap`/`selection`‚Üí`SfxService.playUiTap` keycap tick, `warning`‚Üí`playUiWarn` buzz; `success`/`reward`/`none` silent ‚Äî their moments own audio; `sound: false` opts out a handler-owned button, e.g. quest CLAIM ALL). Core-loop beats: `playSetLogged` in `_logSet` (3 rotated variants + 1s burst cooldown) and `playRestGo` in both rest surfaces' live-finish guards. Assets from `ops/gen_ui_sfx.py` (band-limited C-major squares on a loudness ladder: tap 0.10 < warn 0.20 < set-log 0.22 < rest 0.30 < ceremony). Two toggles: master **Sound** gates everything; the **UI sounds** sub-toggle (`ui_sounds_enabled_v1`, default ON, dims while Sound is off) gates only the tap tick. `SfxService.applyGlobalAudioContext` (boot) sets **audioFocus none + sonification** on ALL audio so app sounds never pause the user's Spotify; the platform layer is skipped entirely under `flutter test` (the `onPlayForTest` recorder is the test contract ‚Äî audioplayers' internal init future can't be caught). Selection-tier ticks on chips are deliberately **not built** (v2, pending core-loop validation).
